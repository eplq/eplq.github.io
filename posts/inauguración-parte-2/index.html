Publicando el blog en GitHub Pages - Inauguración Parte 2

June 24, 2022

Muy buenas a todos de nuevo, en este artículo vamos a publicar ya el blog usando submódulos de Git.

Soy consciente de que se puede hacer con la rama gh-pages, pero me apetece probar los submódulos.

Si no has leído la primera parte, te recomiendo empezar por ahí para tener contexto.

Qué son los submódulos de Git#

Son repositorios dentro de subdirectorios de un repositorio. Git lo maneja de manera que los commits quedan separados y no interfieren entre sí.

Entonces hace mucho más sencilla la sincronización de dos repositorios. Podéis ampliar más información en el libro de Git SCM.

La idea ahora es crear un repositorio nuevo, que en mi caso será eplq/eplq.github.io para que GitHub sepa que ese repositorio tiene que ser publicado en Pages.

Una vez creado, vamos a nuestro repositorio de Hugo y lo añadimos como submódulo:

git submodule add git@github.com:<usuario>/<usuario>.github.io public

Dos apuntes:

  • Recomiendo que se añada el repositorio a través de SSH, para poder tener la capacidad de escribir en el repositorio. Esto requiere que se hayan configurado previamente las claves SSH públicas.
  • La carpeta es public porque es donde Hugo deposita los archivos listos para publicar.

Este segundo artículo lo estoy escribiendo ya en Hugo, y tengo la tranquilidad de que en la cabecera de Front Matter está marcado como borrador, lo que hará que no se publique.

draft: true

Ahora que tenemos Git configurado, vamos a tratar de publicarlo.

Publicando el blog#

Para ello tenemos que sincronizar el repositorio público. Vamos a hacer los commits oportunos.

Lo primero de todo es comprobar que no tengamos nada preparado (staged) en Git. Si es así, hacemos commit o lo restauramos, pero dejamos lo vacío.

Ahora generamos el sitio estático:

hugo

Y lo preparamos:

git add public

Sale lo siguiente:

¿Por qué? Pues porque los cambios se van a subir al repositorio del proyecto completo y no al del submódulo. Por tanto, vamos a hacer un commit:

git commit -m "public actualizado"

git pull
git push

Y ahora subimos los cambios al submódulo, de manera independiente, porque, al fin y al cabo, es un repositorio:

cd public
git add . # preparamos los cambios
git commit -m "inauguración" # los confirmamos
git pull # comprobamos que si hay cambios en el remoto
git push # subimos los cambios

Y con esto ya estaríamos. Ya hemos publicado el blog y ya está disponible en el enlace de GitHub Pages.
